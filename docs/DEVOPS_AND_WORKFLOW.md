# DevOps и работа с проектом

Как правильно работать с общим кодом, Docker, настройками, БД и миграциями.

---

## 1. Работа с общим кодом (Git)

### Базовая схема

- **main** — стабильная ветка (то, что уже зарелизили или готово к деплою).
- Фичи/правки делаются в **отдельных ветках**, потом — merge (или Pull Request) в `main`.

### Типичный цикл

```bash
# Обновить main и начать фичу
git checkout main
git pull origin main
git checkout -b feature/activity-log

# Работа, коммиты
git add .
git commit -m "feat: add activity log for task create/update"

# Отправить ветку и создать PR (или merge локально)
git push -u origin feature/activity-log
```

### Что не коммитить

- Секреты: пароли, JWT secret, OAuth client secret. Только переменные окружения или конфиги вне репо.
- Локальные IDE-настройки (если не общие для команды): часто в `.gitignore`.
- `build/`, кэши, логи.

### Коммиты

- Осмысленные сообщения: что сделано и зачем (например `feat: ...`, `fix: ...`, `chore: ...`).
- Один коммит — одна логическая правка, чтобы было проще откатить или найти причину бага.

## 2. Docker — как им пользоваться

### Роли контейнеров

- **postgres, redis, minio** — инфраструктура. Поднимаются через `docker-compose up -d`.
- **app** (из `docker-compose.app.yml`) — само приложение в образе. Нужно только когда хочешь «всё в контейнерах» или деплой.

### Локальная разработка (рекомендуется)

1. Поднять только инфраструктуру:
   ```bash
   docker-compose up -d
   ```
2. Приложение запускать на хосте:
   ```bash
   ./gradlew bootRun
   ```
   Так быстрее перезапуск и отладка; конфиг в `application.properties` с `localhost` уже подходит.

### Запуск всего в Docker

```bash
docker-compose -f docker-compose.yml -f docker-compose.app.yml up -d --build
```

- Приложение подключается к сервисам по именам: `postgres`, `redis`, `minio` (переменные в `docker-compose.app.yml`).
- Логи: `docker-compose -f docker-compose.yml -f docker-compose.app.yml logs -f app`.

### Остановка и очистка

```bash
# Остановить контейнеры
docker-compose down

# Остановить и удалить тома (БД и данные MinIO/Redis обнулятся)
docker-compose down -v
```

Тома нужны, чтобы данные БД/Redis/MinIO сохранялись между перезапусками. `-v` используй только когда сознательно сбрасываешь окружение.

## 3. Настройки приложения

### Где что лежит

- **Базовые настройки:** `src/main/resources/application.properties`.
- **Профиль prod:** можно завести `application-prod.properties` и включать его через `spring.profiles.active=prod` (или переменная окружения).

### Переменные окружения

Spring Boot маппит переменные в свойства:  
`SPRING_DATASOURCE_URL` → `spring.datasource.url`,  
`JWT_SECRET` → `jwt.secret` и т.д.

Для прода и Docker все секреты и URL лучше задавать через переменные окружения, а не хардкодить в репо.

### Что важно не светить

- Пароли БД, Redis (если есть пароль).
- `jwt.secret`, OAuth client id/secret, ключи MinIO.
- В репозитории — только плейсхолдеры или значения для локального dev (типа `postgres/postgres`).

## 4. База данных и миграции (Flyway)

### Как устроено

- Миграции — SQL-файлы в `src/main/resources/templates/db/migration/`.
- Имена: `V1__описание.sql`, `V2__...`, и т.д. Порядок по номеру.
- При старте приложения Flyway проверяет таблицу `flyway_schema_history` и применяет только недостающие миграции.
- **Важно:** уже применённую миграцию менять нельзя (иначе контроль версий БД сломается). Только новые файлы (V23, V24, ...).

### Добавление новой миграции

1. Узнай последний номер (например V22).
2. Создай файл `V23__short_description.sql`.
3. Пиши только добавочный SQL: `ALTER TABLE ...`, `CREATE TABLE ...`, и т.д.
4. Запусти приложение (или `./gradlew flywayMigrate` если настроен) — миграция применится.

### Локально

- После `docker-compose up -d` PostgreSQL уже запущен; при `bootRun` Flyway отработает сам.
- Если что-то пошло не так — можно сбросить БД (удалить том и поднять заново) и применить миграции с нуля. В проде так не делают — только вперёд, новыми миграциями.

## 5. Как правильно запускать и проверять

### Перед работой

1. `docker-compose up -d` — поднять БД, Redis, MinIO.
2. При необходимости — проверить порты: 5432 (PostgreSQL), 6379 (Redis), 9000/9001 (MinIO).

### Запуск приложения

- Разработка: `./gradlew bootRun` (или `gradlew.bat bootRun` на Windows).
- Проверка: http://localhost:8084/actuator/health, http://localhost:8084/swagger-ui.html.

### Тесты

- Полный прогон (включая контекст Spring и БД): нужна поднятая инфраструктура, затем `./gradlew test`.
- Только сборка: `./gradlew build -x test` или `./gradlew bootJar`.

## 6. Краткий чеклист «перед продом»

- [ ] Все секреты уходят в переменные окружения (или секрет-менеджер).
- [ ] Профиль `prod`: отключён `show-sql`, настроено логирование, при необходимости отключён/ограничен Swagger.
- [ ] CORS настроен на реальный фронт/домен, а не `*`.
- [ ] Миграции применяются при деплое (старт приложения или отдельный шаг в CI/CD).
- [ ] Health-check по `/actuator/health` используется оркестратором (Docker/Kubernetes).
- [ ] Образ приложения собирается из Dockerfile; в проде не запуск из-под root (в текущем Dockerfile уже не root).

Если будешь держать в голове: «общий код — через ветки и осмысленные коммиты; инфра — Docker; настройки и секреты — env; БД — только миграции вперёд», этого достаточно, чтобы уверенно вспомнить и развивать проект.
