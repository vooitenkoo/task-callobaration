# План разработки Task Collaboration (Jira-подобное приложение)

Полномасштабный план: видение продукта, требования, функционал бэкенда, стек, этапы и идеи под пет-проект уровня Middle Java 2026.

---

## 1. Видение продукта

**Task Collaboration** — веб-приложение для командной работы над задачами в стиле Jira/Linear: проекты, доски, задачи, подзадачи, спринты, роли, комментарии, уведомления, отчёты.

**Целевая аудитория:** небольшие команды и личные проекты. Пет-проект должен демонстрировать умение проектировать API, безопасность, современный стек и актуальные практики (в т.ч. опциональное использование AI).

---

## 2. Требования к приложению (что оно должно выполнять)

### 2.1 Функциональные

| Область | Требование | Статус |
|--------|------------|--------|
| **Пользователи** | Регистрация, вход (email/пароль), OAuth2 (Google, GitHub) | ✅ Есть |
| **Пользователи** | Профиль: имя, аватар (MinIO), настройки | ✅ Есть |
| **Пользователи** | Роли: USER, ADMIN (глобально) | ✅ Частично (роль есть) |
| **Проекты** | CRUD проектов, название, описание, ключ (например TASK) | ✅ Есть |
| **Проекты** | Участники: приглашение, роли (OWNER, ADMIN, MEMBER), исключение | ✅ Есть (роли) |
| **Задачи** | CRUD, заголовок, описание, статус, дедлайн, исполнитель, приоритет | ✅ Частично (приоритета может не быть) |
| **Задачи** | Привязка к проекту, связь с другими задачами (parent/child, блокирует) | ✅ Частично (проект есть) |
| **Задачи** | Вложения (файлы), чек-листы | ✅ Файлы есть |
| **Чат/комментарии** | Комментарии к задаче или общий чат по проекту, WebSocket | ✅ Есть |
| **Аудит/активности** | Лог действий по задаче/проекту (кто что создал/изменил) | ⬜ Таблица есть, API/логика — нет |
| **Уведомления** | Уведомления о назначении, комментариях, смене статуса (in-app, email, push) | ⬜ Нет |
| **Спринты/доски** | Спринты, доски (Kanban), перенос карточек по статусам | ⬜ Нет |
| **Поиск и фильтры** | Поиск по задачам/проектам, фильтры по статусу, исполнителю, дате | ⬜ Частично (списки по проекту) |
| **Отчёты и аналитика** | Дашборд: задачи по статусам, по пользователям, бурндаун и т.д. | ⬜ Нет |
| **API** | REST, документация (OpenAPI/Swagger), версионирование (опционально) | ✅ REST + OpenAPI |

### 2.2 Нефункциональные

- **Безопасность:** аутентификация (JWT + OAuth2), авторизация по ролям и членству в проекте, без хранения паролей в открытом виде.
- **Масштабируемость:** stateless API, возможность горизонтального масштабирования, кэш (Redis) где уместно.
- **Надёжность:** миграции БД через Flyway, консистентность данных, обработка ошибок и валидация.
- **Наблюдаемость:** логи, health-check (Actuator), метрики (опционально Prometheus).
- **Современность:** Java 17+, Spring Boot 3.x, актуальные версии библиотек, при необходимости — виртуальные потоки, реактивность, AI-интеграции.

## 3. Полный функционал бэкенда (список того, что предстоит сделать)

Ниже — список возможностей с пометкой «есть» / «сделать» / «улучшить».

### 3.1 Аутентификация и пользователи

- [x] Регистрация (email + пароль), хэш BCrypt.
- [x] Вход (email + пароль), выдача JWT (access + refresh).
- [x] OAuth2: Google, GitHub; создание/обновление пользователя, выдача JWT после входа.
- [ ] **Refresh token:** эндпоинт `POST /api/auth/refresh` (обмен refresh на новую пару токенов).
- [ ] **Выход:** инвалидация refresh (blacklist в Redis или хранение выданных refresh).
- [ ] **Восстановление пароля:** запрос сброса по email, ссылка с токеном, смена пароля.
- [ ] **Профиль:** смена имени, аватара, пароля, привязка/отвязка OAuth2.

### 3.2 Проекты и участники

- [x] CRUD проектов, участники с ролями (OWNER, ADMIN, MEMBER).
- [ ] **Приглашения:** ссылка или email-приглашение в проект, принятие/отклонение.
- [ ] **Ключ проекта:** короткий ключ (например TASK) для нумерации задач (TASK-1, TASK-2).
- [ ] **Настройки проекта:** дефолтные статусы, типы задач, права по умолчанию.

### 3.3 Задачи

- [x] CRUD задач, статус, дедлайн, исполнитель, проект, вложения (файлы).
- [ ] **Подзадачи (subtasks):** иерархия parent/child, прогресс по подзадачам.
- [ ] **Приоритет:** LOW, MEDIUM, HIGH, CRITICAL.
- [ ] **Тип:** TASK, BUG, STORY, EPIC и т.д.
- [ ] **Связи:** блокирует/заблокирована, дубликат, родительская задача.
- [ ] **Чек-листы:** пункты с галочкой внутри задачи.
- [ ] **Метки (labels)/теги:** много ко многим с задачами.
- [ ] **Нумерация:** сквозная в проекте (PROJ-1, PROJ-2) на основе ключа проекта.

### 3.4 Комментарии и общение

- [x] Сообщения по проекту (чат), WebSocket.
- [x] REST для истории сообщений.
- [ ] **Комментарии к задаче:** отдельная сущность, уведомления упомянутым и исполнителю.
- [ ] **Упоминания (@user):** парсинг в комментариях, уведомления.

### 3.5 Активности и аудит

- [x] Таблица `activities` (user, task, type, description, created_at).
- [ ] **Сервис и API:** запись событий при создании/обновлении/удалении задач и ключевых сущностей.
- [ ] **Эндпоинты:** лог по задаче, по проекту, с фильтрами и пагинацией.

### 3.6 Уведомления

- [ ] **Модель:** уведомление (user, type, entity_id, read, created_at).
- [ ] **События:** новая задача, назначение, комментарий, смена статуса, приглашение в проект.
- [ ] **Доставка:** in-app (список + WebSocket или long polling), опционально email (Spring Mail), push (Firebase/Web Push).
- [ ] **Настройки:** какие типы уведомлений получает пользователь.

### 3.7 Спринты и доски

- [ ] **Спринты:** даты начала/окончания, цели, привязка задач к спринту.
- [ ] **Доска Kanban:** колонки = статусы проекта, перенос карточек (смена статуса), порядок в колонке.
- [ ] **Бэклог:** задачи без спринта, перетаскивание в спринт.

### 3.8 Поиск и фильтры

- [ ] **Поиск:** по названию/описанию задач, по проектам (полнотекстовый или LIKE).
- [ ] **Фильтры:** по статусу, исполнителю, приоритету, меткам, дате, проекту; комбинирование.
- [ ] **Сохранённые фильтры:** пользовательские наборы фильтров (название, JSON критериев).

### 3.9 Отчёты и аналитика

- [ ] **Дашборд:** количество задач по статусам, по исполнителям, просроченные.
- [ ] **Бурндаун/прогресс спринта:** если есть спринты.
- [ ] **Экспорт:** CSV/Excel по задачам (опционально).

### 3.10 Инфраструктура и DevOps

- [x] Docker Compose: PostgreSQL, Redis, MinIO.
- [x] Dockerfile приложения, docker-compose для полного запуска.
- [ ] **Очереди (RabbitMQ/Kafka):** асинхронная обработка уведомлений, событий, тяжёлых отчётов.
- [ ] **Кэш:** Redis для сессий/blacklist refresh, кэш частых запросов (списки проектов, участников).
- [ ] **Метрики:** Prometheus + Grafana или Micrometer.
- [ ] **Логирование:** структурированные логи (JSON), уровни по профилям.
- [ ] **CI/CD:** GitHub Actions (сборка, тесты, образ, деплой на облако или VPS).

## 4. Современные технологии и практики (уровень Middle 2026)

Что имеет смысл использовать или изучить в рамках этого проекта:

- **Java 21 LTS** — переход с 17 на 21 (Virtual Threads, улучшения в языке).
- **Spring Boot 3.2+** — Virtual Threads (`spring.threads.virtual.enabled=true`), последние фичи Security.
- **Очереди сообщений:** RabbitMQ или Kafka — события (задача создана, комментарий), уведомления, фоновые задачи.
- **Кэш:** Redis — blacklist JWT/refresh, кэш списков, сессии при необходимости.
- **Поиск:** Elasticsearch или OpenSearch — полнотекстовый поиск по задачам (опционально).
- **Тесты:** JUnit 5, Mockito, Testcontainers (PostgreSQL, Redis), интеграционные тесты API, при необходимости — тесты WebSocket.
- **Документация API:** SpringDoc OpenAPI 2.x, описание сценариев, примеры запросов.
- **Безопасность:** JWT с коротким временем жизни access, refresh в httpOnly cookie или отдельный эндпоинт; rate limiting (Bucket4j или Resilience4j).
- **Мониторинг:** Micrometer + Prometheus, health, readiness, метрики по эндпоинтам.

## 5. Где и как использовать AI (идеи)

- **Генерация описаний/резюме:** по названию задачи или по логу комментариев — краткое описание (вызов внешнего API: OpenAI, Claude, локальная модель).
- **Умные подсказки:** предложение исполнителя по тексту задачи или истории (embedding + похожие задачи).
- **Классификация:** автоматические метки или приоритет по тексту задачи (модель или API).
- **Чат-бот в комментариях:** ответы на вопросы по задаче/проекту (RAG по контексту проекта).
- **Резюме спринта:** автоматический отчёт «что сделано» по задачам и комментариям.

Реализация: отдельный сервис или слой, вызовы асинхронные (очереди), без блокировки основного API; логирование и лимиты запросов.

## 6. Этапы разработки (предлагаемая очередность)

1. **Безопасность и база** — убедиться, что auth/authz работают корректно, добавить refresh token и (опционально) logout (см. SECURITY.md).
2. **Задачи и проекты** — приоритеты, подзадачи, ключ проекта, нумерация; приглашения в проект.
3. **Активности** — сервис записи событий, API лога по задаче/проекту.
4. **Уведомления** — модель, создание при событиях, in-app API, WebSocket; затем email/push при желании.
5. **Очереди** — RabbitMQ/Kafka для событий и уведомлений, асинхронная обработка.
6. **Спринты и доски** — спринты, доска Kanban, бэклог.
7. **Поиск и фильтры** — расширенные фильтры, сохранённые, при необходимости полнотекстовый поиск.
8. **Отчёты и дашборд** — агрегации по задачам, простой дашборд.
9. **AI-фичи** — одна-две идеи из раздела выше (например, генерация описания, теги).
10. **Инфра и прод** — метрики, логи, CI/CD, деплой.
